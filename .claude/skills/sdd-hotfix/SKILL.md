# Hotfix 工作流

## 触发

用户说 `/sdd-hotfix` 或「紧急修复」或「线上出问题了」

## 和 fix-bug 的区别

- **fix-bug**：完整流程，有时间做彻底修复
- **hotfix**：生产环境紧急问题，优先恢复服务，代码质量可以后续优化

## 流程

### Phase 1: 止血

1. 确认问题影响范围和严重程度
2. 如果可以快速回滚，先执行 `/sdd-rollback-change`
3. 如果不能回滚，进入 Phase 2

### Phase 2: 最小修复

4. 定位根因（快速，不求完美）
5. 编写最小修复代码（只修复问题，不重构）
6. 手动验证修复有效

### Phase 3: 提交

7. 从 `main` 创建 `hotfix/<简短描述>` 分支
8. 提交：`fix(<scope>): [HOTFIX] <描述>`
   - body 说明问题现象、影响范围、临时修复方案
9. 合并到 `main`

### Phase 4: 后续（可以稍后做）

10. 创建 FR 记录需要的彻底修复
11. 在后续 sprint 中用 `/sdd-fix-bug` 做彻底修复
12. 补充回归测试

## 约束

- Hotfix 代码量应尽可能小
- 不做任何「顺手」的改动
- 后续必须补做彻底修复（记录为 FR）
