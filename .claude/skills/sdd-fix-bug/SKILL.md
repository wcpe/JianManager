# Bug 修复工作流

## 触发

用户说 `/sdd-fix-bug <描述>` 或「修复这个 bug」或「这里有问题」

## 流程

### Phase 1: 复现

1. 理解 bug 描述（用户报告、错误日志、截图）
2. 确定复现步骤
3. 确定影响范围（哪个模块、哪些 FR 受影响）

### Phase 2: 定位

4. 分析根因：
   - 读取相关代码（router → service → model）
   - 检查相关 API Spec 是否和实现一致
   - 检查是否违反了 ARCHITECTURE.md 中的约束
5. 确认根因后，判断：
   - 是代码 bug → Phase 3
   - 是 spec 设计问题 → 更新 spec，再 Phase 3
   - 是 ADR 需要变更 → 创建新 ADR，再 Phase 3

### Phase 3: 修复（带重试循环）

6. 编写修复代码
7. 编写回归测试（防止再次出现）
8. 验证修复：
   - 运行相关测试
   - 运行 `go build ./...` 确认编译通过
   - 手动验证修复效果
9. **如果验证失败**：
   - 记录当前方案为什么失败
   - 分析失败原因（根因判断错误？修复方案有副作用？）
   - 回滚当前修复代码
   - **回到步骤 6，换一个方案重新修复**
10. 最多重试 3 次。3 次都失败后：
    - 输出所有尝试过的方案和失败原因
    - 暂停，向用户报告需要人工介入：
      ```
      ⚠️ 已尝试 3 种修复方案均未成功：
      方案 1: <描述> — 失败原因: <原因>
      方案 2: <描述> — 失败原因: <原因>
      方案 3: <描述> — 失败原因: <原因>
      
      需要你提供更多上下文或确认根因方向。
      ```

### Phase 4: 收尾

11. 修复验证通过后：
    - 运行 `go vet ./...` 确认无新问题
    - 检查修复是否需要文档同步（通常不需要，除非改了 API 行为）
12. 提交：`fix(<scope>): 修复<bug描述>`
    - body 中说明根因和修复方案
    - 如果经历了重试，说明「尝试了 X 种方案，最终用 Y 方案解决，因为 Z」

## 约束

- Bug 修复不得引入新功能（范围纪律）
- 必须有回归测试
- 提交 message 中必须说明根因
- 每次重试必须先回滚上一次的修复代码，保持工作区干净
- 3 次重试上限，防止无限循环
