# 重构工作流

## 触发

用户说 `/sdd-refactor-code` 或「重构 XXX」

## 前提

- 重构不得改变外部行为（API 不变、数据库不变、用户可见行为不变）
- 如果需要改变外部行为，那是 feature，不是 refactor

## 流程

### Phase 1: 定义范围

1. 明确重构目标（提取接口？拆分大文件？优化性能？）
2. 确认影响范围（哪些文件会变）
3. 确认重构不违反任何 ADR

### Phase 2: 准备

4. 确保当前测试全部通过（基线）
   ```bash
   go test ./...
   ```
5. 如有必要，先补充缺失的测试（重构前的保护网）

### Phase 3: 重构

6. 小步重构，每步可验证
7. 每步后运行测试确认无回归
8. 不得在重构过程中添加新功能

### Phase 4: 验证

9. 全量测试通过
10. `/sdd-review` 检查是否引入了 spec 不一致
11. 手动验证关键路径行为不变

### Phase 5: 提交

12. 提交：`refactor(<scope>): <描述>`
    - body 说明重构动机和方式
    - 不得混入功能变更

## 约束

- 单次重构只做一件事（不混合多种重构）
- 重构和功能变更必须分开提交
- 如果重构影响了架构分层，需要同步更新 `docs/ARCHITECTURE.md`
- 如果重构涉及 ADR 记录的决策，需要检查是否仍然对齐
