# 需求转 FR

## 触发

用户说「我需要 XXX」「新增需求：XXX」「加个 XXX 功能」「有个想法：XXX」

或 `/sdd-new-requirement <需求描述>`

## 流程

### Step 1: 理解需求

用户输入可能是：
- 一句话描述：「我需要支持多语言」
- 多条需求：「需要定时备份、告警通知、还有模板管理」
- 模糊描述：「玩家邮箱那个功能」
- 带上下文：「参考 JianAgent 的插件桥，做一个类似的功能」

如果需求不清晰，**向用户提问**（一次一个问题）：
- 这个功能面向谁？（管理员 / 组管理员 / 组成员）
- 和现有哪个功能相关？
- 有没有参考实现？（JianAgent / MCSManager / 其他项目）

### Step 2: 判断类型

每个需求判断是 feat 还是 fix：

| 判断依据 | 类型 |
|---|---|
| PRD.md 中不存在相关 FR | feat（新功能） |
| PRD.md 中已有 FR 但实现有问题 | fix（修复） |
| PRD.md 中已有 FR 但功能不完整 | feat（增强，新 FR） |
| 用户明确说「修」「bug」「不工作」 | fix |

不确定时向用户确认。

### Step 3: 确定元数据

对每个 FR 确定：

- **编号**：PRD.md 中最大 FR 编号 + 1（如最大是 FR-016 → 新的从 FR-017 开始）
- **优先级**：
  - P0：核心功能，没有它系统不能用
  - P1：重要功能，影响用户体验
  - P2：增强功能，锦上添花
- **状态**：新需求统一为 `📋 todo`
- **依赖**：是否依赖其他已有的 FR（需要那个 FR 完成后才能开始）
- **关联 ADR**：如果需求涉及架构决策，标注相关 ADR

### Step 4: 写入 PRD.md

将 FR 追加到 PRD.md 对应的优先级区域：

```markdown
### FR-017: <功能名称>
- **状态**: 📋 todo
- **优先级**: P0/P1/P2
- **描述**: <一句话描述>
- **验收标准**:
  - [ ] <可测试的条件 1>
  - [ ] <可测试的条件 2>
  - [ ] <可测试的条件 3>
- **关联 ADR**: <如有>
- **关联 API**: `<如能预判 endpoint>`
- **依赖**: <如依赖其他 FR>
```

验收标准要求：
- 每条必须可测试（用户能看到、能操作、能验证）
- 不写「支持 XXX」这种模糊描述，写「用户可以 XXX，系统返回 XXX」
- 3-7 条，覆盖 happy path + 主要 error case

### Step 5: 向用户展示结果

```
已将需求转化为以下 FR：

FR-017: 定时备份
  类型: feat | 优先级: P1 | 状态: 📋 todo
  验收标准: 4 条
  依赖: FR-005（实例生命周期）

FR-018: 告警通知
  类型: feat | 优先级: P1 | 状态: 📋 todo
  验收标准: 5 条
  依赖: FR-010（监控指标）

已写入 docs/PRD.md。

接下来要并行开发吗？
  → /sdd-parallel-develop FR-017 FR-018
  → 或串行: /sdd-develop-feature FR-017
```

### Step 6: 提交文档

```
git add docs/PRD.md
git commit -m "docs: 新增 FR-017 定时备份, FR-018 告警通知"
```

## 约束

- 只做需求分析和 FR 定义，不写代码
- 不修改已有 FR 的内容（只追加新 FR）
- 如果需求和已有 FR 重叠，向用户指出并建议合并或区分
- 验收标准必须是用户视角（不是技术实现视角）
