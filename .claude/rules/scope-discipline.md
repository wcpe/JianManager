# 范围纪律

> 只做当前 spec 定义的事，不做「顺手」发现的事。

## 规则

1. **严格对齐 spec**：实现的功能必须和 `docs/specs/<feature>/api.md` 一致，不多不少
2. **不顺手重构**：发现旧代码有问题时，记录到 backlog，不在当前 feature 中修复（除非阻塞当前 feature）
3. **不顺手加功能**：「这个也顺便做了吧」是范围蔓延的源头。记录为新 FR，不在此处实现
4. **不顺手优化**：「这段代码可以优化」如果不是当前 feature 的目标，不改

## 处理发现的问题

实现过程中发现的超出范围的问题：

1. 在代码中添加 `// TODO(FR-XXX): <描述>` 注释（不阻塞当前工作）
2. 或在 `docs/PRD.md` 中追加新 FR（状态 `📋 todo`）
3. 不在当前 feature 的 commit 中混入无关变更

## 例外

以下情况允许在当前 feature 中处理：

- 当前 feature 的实现依赖这个变更（阻塞关系）
- 一个变量重命名等极小改动且和当前代码紧密耦合
- 安全漏洞（必须立即修复）

## 检查

`/sdd-review` 会检查 diff 中是否有超出 spec 范围的变更。
