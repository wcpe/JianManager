# ADR-024: 全文索引后台化首建

- **日期**: 2026-06-24
- **状态**: accepted
- **部分取代**: ADR-017 决策 §3「构建在 Worker 进程内同步进行」——首建改为**后台异步**；ADR-017 其余决策（引擎、落位、增量按指纹、忽略规则、gRPC 转发）不变。
- **上下文**: FR-113。ADR-017 决策 §3 规定「某实例首次被搜索时全量构建一次……构建在 Worker 进程内**同步**进行」。真机走查发现：大工作目录（成百上千文件、动辄数百 MB）首次搜索时，`SearchFiles` 同步全量读取并切词所有文本文件，耗时可达数秒到数十秒，期间 gRPC 调用与前端 UI 阻塞，体验不可接受。增量查询（指纹比对、仅重读变更文件）本就很快，问题专在**首建**这一次重活。需在不破坏 ADR-017 其余决策的前提下，把首建移出查询关键路径，并向前端表达「索引中」进度。

## 决策

### 1. 每实例索引就绪状态（进程内，重启即重建）

`search.Index` 增加两个原子标志与一个完成信号：

- `ready atomic.Bool`：是否已完成至少一次构建（可信查询）。进程内状态，**不落盘**——Worker 重启后归零，首查重新后台构建（构建内 `load()` 复用落盘索引做快速追平，非从零）。
- `building atomic.Bool`：是否有后台构建在途。CAS `false→true` 保证**单飞**（同一实例至多一个构建 goroutine，避免重复全量扫描与对完成信号的重复关闭）。
- `builtCh chan struct{}`：首建完成（成功或失败）时关闭，供有界等待者唤醒。

### 2. 查询路径：未就绪后台构建 + 小目录快路径

`SearchFiles` RPC 处理：

- **已就绪**：同步增量 `Update`（指纹比对，快）+ 查询，返回命中 + `indexing=false`。与 ADR-017 既有行为一致（保留「查询前一次 stat 遍历」的可接受代价）。
- **未就绪**：CAS 启动一次后台全量构建（`EnsureBuilding`，非阻塞）；随后**有界等待** `WaitReady(200ms)`：
  - 构建在预算内完成（小目录，毫秒级）→ 视为就绪，本次即同步查询出结果（`indexing=false`）。小目录无「索引中」往返、不退化、既有同步语义不破。
  - 预算内未完成（大目录）→ 立即返回 `indexing=true`、空命中，**不碰索引锁、不阻塞**。前端据此显示「索引中」并自动重试。

200ms 预算是「小目录快路径」与「大目录不卡」的平衡点：远大于小目录构建耗时（微秒级，含 goroutine 调度裕量），又远小于人对卡顿的感知阈值。

### 3. 构建容错：成功或失败都置就绪（自愈）

后台构建 goroutine 无论 `Update` 成功或返回顶层错误（如 workDir 被并发删除），完成时都置 `ready=true` 并关闭 `builtCh`。理由：若失败保持未就绪，则每次查询都重新 CAS 构建 → 永久「索引中」死循环。置就绪后，后续查询走同步增量 `Update`，其自带的逐文件容错与下次扫描会自然追平（自愈）。构建顶层错误记日志，不向调用方报错（首查仍返回空命中 + `indexing=false`，与「无匹配」一致）。

### 4. 契约与前端

- **proto**：`SearchFilesResponse` 加性新增 `bool indexing = 3`（protoc 重新生成，不手改 pb.go）。
- **CP**：`service.SearchResult` 加 `Indexing bool`，`FileService.SearchFiles` 透传 `resp.Indexing`；CP 仍仅转发、不持有索引（守 ADR-017 §5）。
- **前端**：`SearchPanel` 收到 `indexing=true` 时显示「索引中（首次建立索引，请稍候）」并自动重试同一查询（带请求序号防抖竞态），直到 `indexing=false` 出结果；用户改查询则旧重试作废。

## 理由

- **首查不卡**：重活（全量读+切词）移出查询关键路径，大目录首查立即返回进度而非冻结。
- **小目录不退化**：有界快路径让小目录首查仍同步出结果，既有同步语义与既有测试不破。
- **零新增依赖**：仅 `sync/atomic` + goroutine + channel，契合项目便携自包含取向（ADR-017 §引擎决策一脉相承）。
- **进程内状态、重启自愈**：就绪态不落盘，Worker 重启后首查重建（复用落盘索引快速追平），无需持久化就绪态、无状态迁移负担。
- **单飞 + 容错**：CAS 防重复构建；成功/失败都置就绪防「索引中」死循环。

## 后果

- Worker 重启后，每实例首次搜索可能返回一次 `indexing=true`（前端自动重试，通常下一拍即出结果）。这是进程内就绪态的预期代价。
- 小目录首查引入至多 200ms 的有界等待预算（实际构建远快于此，几乎无感）。
- 构建顶层错误被吞（仅记日志）以避免「索引中」死循环；查询正确性由后续同步增量 `Update` 的逐文件容错兜底。
- `Index` 新增进程内字段，不改落盘 `index.gob` 格式（`indexFormatVersion` 不变），与既有落盘索引前后兼容。

## 替代方案

- **保持同步首建（ADR-017 §3 原样）**：大目录首查阻塞，违反 FR-113 验收「大目录首查不卡 UI」；否决。
- **定时/建服时预建索引**：需后台调度器与生命周期钩子，复杂度高；惰性后台首建已满足验收，YAGNI。
- **就绪态落盘持久化**：省去重启后一次重建，但增加落盘格式与一致性负担；重启重建本就复用落盘索引快速追平，收益不抵复杂度；否决。
- **首查无界等待构建完成**：等于没后台化，仍阻塞；否决。
