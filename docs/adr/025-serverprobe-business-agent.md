# ADR-025: ServerProbe 从监控探针进化为 JM 全能业务 agent

- **日期**: 2026-06-24
- **状态**: accepted（骨架；实现细节随 FR-117 落地）
- **上下文**: JBIS 业务对接平台（FR-115~127）。运营者要 JM 成为顶层总管，穿透到每个服务端的每个业务插件读写数据。需要游戏服内有一个 JM 的 agent 既采监控、又对接业务插件。现有 ServerProbe 已是「监控探针」（反向 WS 桥 + 只读采集 + 轻量治理，ADR-016），且其子模块自带红线「探针主体只读、不引数据库、core/api 不碰平台 API」。问题：再起一个独立业务插件（JianAgent），还是让 ServerProbe 长出业务能力？

## 决策

**一个 ServerProbe = 本服务端唯一全能 agent，对外统一、对内分层。**

1. **对外统一**：单插件分发、单反向 WS 连接、单身份/token。符合"总管的唯一触手"定位。
2. **对内分层 + 事故域隔离**（铁律）：
   - 监控层（既有 core/api）：纯只读，保持纯净，不碰业务。
   - 业务对接层（新增，落 **platform 层**）：BusinessHost + per-plugin Provider；**独立线程池 + 独立异常边界 + 守护式初始化**。业务 Provider 卡死/抛异常/插件缺失，只降级该域命令回执，**监控采集与桥心跳完全不受影响**。
3. **守 ServerProbe 自身红线**：业务代码只落 platform 层，core/api 保持只读纯净、不引数据库、不碰平台 API。
4. **拒绝独立 JianAgent 插件**：两个 Bukkit 插件同 JVM、共享主线程/堆/进程，并不带来更强运行时隔离；真正的隔离来自线程池+异常边界（一个插件分模块即可达成），独立插件徒增双连接/双身份/双发版的运营面。

## 理由
- 一个 agent/连接/身份契合"唯一触手"愿景，运营面最简。
- 事故域隔离是代码级纪律（线程/异常分离），非独立插件要求；同 JVM 下独立插件不提供进程/内存隔离。
- platform 层承接业务、core/api 保持纯净，两全 ServerProbe 红线与业务诉求。

## 后果
- ServerProbe platform 层新增业务对接模块（FR-117 骨架 + FR-118/120/125 各域 Provider）。
- ServerProbe 子模块需在其自身仓追加一条 ADR 记录"监控探针→业务 agent"定位演进。
- 业务模块必须守护式初始化：其初始化失败降级为"业务不可用"，不得拖垮整个探针（监控继续）。

## 关系

- **ADR-026**：业务对接层采用"适配器 + manifest"而非让插件实现 SPI。
