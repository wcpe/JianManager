# ADR-035: 导播台 WS 预热并发模型（并发上限 + 非激活降频）

- **日期**: 2026-06-27
- **状态**: accepted（随 FR-168 落地）
- **上下文**: FR-166/167 把工作区做成可组合卡片画布 + 跨实例预设。FR-168 导播台要在多个预设（场景）间**瞬切零延迟**（OBS 式）。要零延迟，目标场景的卡（终端 / 监控等）必须**已建好 WS 连接并保活**；若切到才连，会有数秒等待。但「多场景同时保活 + 全速渲染」会**过载浏览器**：①WS 连接数受限（HTTP/1.1 同域 ~6，WS 类似），多实例多场景轻易撑爆；②多 xterm / 图表同时全速重绘吃满 CPU。design §9 明确点出此约束：「导播台预热=多实例 WS 连接保活；**必须设并发上限**，非当前场景卡片降频/暂停渲染，切换才全速，防浏览器过载」。

## 决策

**导播台用「有上限的 WS 保活连接池 + 非激活场景降频/暂停渲染 + LRU 驱逐」的预热并发模型。**

1. **场景状态机**：每个场景（= FR-167 预设）处于三态之一——**激活**（全速渲染 + WS 全速）/ **预热**（WS 保活，但渲染降频/暂停）/ **未预热**（无连接）。**激活唯一**；预热是一个**受上限约束的集合**。
2. **并发上限（硬约束）**：预热场景数（或保活 WS 连接数）设**可配上限**（默认保守值，如 3~4 场景）。新预热超限 → **LRU 驱逐**最久未激活的预热场景（降为未预热，下次切换时重连）。
3. **非激活降频/暂停渲染**：仅激活场景全速重绘；预热但非激活的场景**暂停/降频渲染**（xterm 停 render 但 WS 继续收数据进缓冲；监控降低或暂停轮询），切到时一次性 flush + 转全速。
4. **瞬切**：切换 = 目标场景（已预热则零延迟）置激活、原激活降为预热；快捷键 / 缩略图点击触发。轮播 = 定时器按序触发瞬切。
5. **纯前端**：仅浏览器侧 WS 连接管理 + 渲染节流，不改后端、不新增进程边界（守架构不变量）。

## 理由
- 零延迟瞬切的唯一办法是预热（提前保活 WS）；但不设上限/不降频必然过载——二者是一体两面，必须同时做。
- LRU 驱逐让「常用几个场景」常驻预热、长尾场景按需重连，平衡体验与资源。
- 状态机把「激活/预热/未预热 × 全速/降频」显式化，避免连接泄漏与隐性全速渲染。

## 后果
- `stores/workspace.ts` 扩或新 `stores/director.ts`：导播台状态机 + WS 连接池 + 并发上限 + LRU。
- 既有终端/监控卡需支持「保活但降频/暂停渲染」模式（xterm 可停 render、监控可降频）。
- 并发上限默认值保守，可配；真机多连接压测为硬验收维度。
