# ADR-011: 制品库——类型分区的内容寻址与资产模型

- **日期**: 2026-06-19
- **状态**: accepted
- **上下文**: 平台要下载并复用 MC 核心 jar，后续还有插件、图片、视频、媒体 blob 等各类二进制资产，需要完整性校验与后续归档能力。纯文件夹缓存（按文件名）无去重无校验；纯扁平哈希池（不分类型）则不利于「按类型/冷热归档、整类备份、切换存储后端」。
- **决策**:
  1. **类型分区 + 内容寻址（CAS）**：资产存 `var/artifacts/<type>/<sha256 前 2 位>/<sha256>.<ext>`；类型内按 sha256 去重，类型间物理分目录。`type` ∈ `core | plugin | image | video | archive | blob`。
  2. **DB 索引 `assets`**：`type`/`name`/`version`/`filename`/`sha256`/`md5`/`size`/`content_type`/`source_url`/`metadata`(JSON)/`storage_state`/`storage_backend`/`ref_count`/`created_at`/`last_used_at`。sha256 既是寻址键也是去重键。
  3. **完整性**：入库即算 sha256+md5；来源提供校验和则比对，不符拒收。
  4. **DB 驱动的生命周期/归档**：按 类型/冷热/占用 选择性归档；归档或外置只改 `storage_state`+`storage_backend`+位置，DB 与引用（sha256）不变；未来可按类型切换存储后端（本地 ↔ 对象存储）而不动上层。
  5. **引用保护**：被模板/实例引用的资产删除前拒绝（`ref_count`）。
- **理由**:
  - 跨类型去重无价值，类型分区零损失，且让浏览/归档/备份/分层都干净（回应「归档难处理」之虑）。
  - 内容寻址天然去重、不可变、自校验；DB 索引提供类型化命名视图与生命周期。
  - core 先落地，模型同样容纳插件/图片/视频/媒体，后续库直接复用，不返工。
- **后果**:
  - 取代 FR-044 早期「cache/cores」表述；核心 jar 升级为制品库第一类资产（FR-045）。
  - FR-034 建服从制品库取核心；FR-014 模板可引用制品；后续插件管理（FR-103）、媒体库复用本库。
  - 归档策略与外置存储后端为后续 FR，本 ADR 只立模型与布局留位。
- **替代方案**:
  - 纯文件夹缓存（按名字）— 无去重 / 无完整性 / 易碰撞，否决。
  - 纯扁平 CAS（不分类型）— 归档 / 分层 / 浏览困难，否决。
  - 直接上外部对象存储 — 便携自包含场景过重，作为未来可选后端而非默认。
