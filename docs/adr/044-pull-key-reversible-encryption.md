# ADR-044: 拉取密钥可逆加密存储 + 管理员可查看（修订 ADR-022 决策①的「只存哈希」）

- **日期**: 2026-06-28
- **状态**: accepted
- **修订**: [ADR-022](022-client-manifest-trust-and-public-endpoint.md) 决策① 的**存储部分**（「落库只存 SHA-256 哈希、明文不可二次读取」→「哈希用于鉴权 + 可逆加密副本供管理员查看」）。ADR-022 其余决策与 `accepted` 状态不变。

## 上下文

客户端 OTA（[ADR-021](021-client-distribution-jvm-updater.md) / [ADR-022](022-client-manifest-trust-and-public-endpoint.md)）的 per-channel **拉取密钥**当前**落库只存 `KeyHash`（SHA-256）+ `KeyPrefix`**，明文仅在创建/轮换时一次性返回、不可二次读取（ADR-022 决策①，同构 JM 既有运行时密钥惯例）。

这一「只存哈希」对**运行时密钥**（如节点 enroll token、JWT secret）是对的：它们由机器持有、可随时重置、泄露面集中在服务端。但拉取密钥是一类**特殊凭据**——它**随客户端整合包发到每个玩家机器**，整合包发出后**所有后续更新都依赖这把密钥**：

- 运营若忘记密钥（只在创建时见过一次明文），**无法为新加入的玩家重建 `jm-updater.json`**（缺 key 客户端拉不到 manifest）。
- 唯一补救是轮换，但轮换会让**所有已分发的老客户端集体失效**（旧 key 被换掉）——等于已分发玩家集体断更，运营灾难。

**关键事实**：ADR-022 决策①自己白纸黑字写明拉取密钥**半公开**（「随整包分发必泄露」「不作内容可信依据，防篡改全靠 manifest 签名」）。既然 key 本就半公开、其真实信任级别就不高，「只存哈希」并没换来多少实质安全，却造成上述运营灾难。让密钥**可查看**，恰好与它的真实信任级别一致。

## 决策

1. **拉取密钥改可逆加密存储**：`ClientPullKey` 新增加密文列 `KeyEnc`。创建/轮换时**同时**写：
   - `KeyHash`=SHA-256(明文)——**鉴权仍只用哈希比对，行为完全不变**（决策①的鉴权语义不动）；
   - `KeyEnc`=AES-256-GCM 加密的明文（含随机 nonce），供管理员事后查看。

2. **加密方案 AES-256-GCM、密钥经 env 注入不入库**：对称加密密钥取自环境变量 `JIANMANAGER_CLIENT_KEY_ENC_SECRET`（32 字节，base64 编码），**同构签名私钥的注入惯例**（敏感信息经 `${ENV}` 注入、不硬编码、不入库，见 `config-files` 规范）。每次加密用 `crypto/rand` 生成独立 12 字节 nonce，密文格式 = base64(nonce ‖ ciphertext+tag)。AEAD（GCM）同时保完整性，篡改密文解密即失败。

3. **未配置加密密钥时优雅降级，绝不阻断建密钥**（与 [ADR-038](038-signer-missing-degraded-startup.md) 降级哲学一致）：
   - `dev_mode=true` 且未注入 → 回退**固定的内置 dev 加密密钥**（仅零配置开发用，源码公开，明示不得用于生产）；
   - `dev_mode=false`（生产）且未注入 → **不写 `KeyEnc`**（密钥照常创建、鉴权照常工作，只是不可查看），查看端点对这类密钥返回业务码 `KEY_NOT_REVEALABLE`。
   - **理由**：拉取密钥的创建/轮换是高频运营动作，把它卡在「必须先配一个对称密钥」会把运维卡死；而签名私钥 fail-closed 是因为它是**信任根**（伪造即投毒），拉取密钥半公开、不是信任根，缺加密密钥时降级为「不可查看」是可接受的，不影响 OTA 安全。

4. **查看端点 `GET /client-channels/:id/keys/:keyId/reveal`**：**仅平台管理员**（RBAC 同频道管理），命中即写审计 `client_key.reveal`（detail 仅元数据：channelId/keyId/keyPrefix，**绝不含明文**）。
   - 该密钥有 `KeyEnc` → 解密返回 `{ "key": "<明文>" }`；
   - 无 `KeyEnc`（存量老密钥 / 创建时未配加密密钥）→ 404 业务码 `KEY_NOT_REVEALABLE`。

5. **迁移边界——存量老哈希密钥不可查**：哈希单向，存量「只有 `KeyHash`」的密钥**无法找回明文**（数学上不可能）。迁移加性（`KeyEnc` 默认空），**仅新建/改值过的密钥**有 `KeyEnc` 可查看。前端对无 `KeyEnc` 的行**禁用「查看」并提示**「此密钥创建于可查看功能之前或未配置加密，不可找回；可『编辑』改为已知值后即可查看（注意改值会使持旧值的客户端失效）」。

## 实施细化（2026-06-28，验收后用户纠正方向）

用户验收后明确：拉取密钥**发出后永久使用**（随整合包分发到每个玩家、所有后续更新都依赖它）。据此细化本 ADR 的运营操作面（决策与信任模型不变，仅操作面调整）：

1. **删除「轮换」**：移除轮换 service `RotateKey` + 端点 `POST .../keys/:kid/rotate` + 前端轮换入口。轮换换 key 会让已分发客户端集体断更，与「永久使用」矛盾——原本就是本 ADR 上下文要消除的运营灾难，不应再以「轮换按钮」诱导运营误操作。
2. **新增「可编辑密钥值」**：管理员可**手动设/改**这把永久 key——创建时 `value` 可填自定义值（留空自动生成）；既有密钥经 `PUT .../keys/:kid`（值 + 名称，平台管理员 + 审计 `client_key.update`）改值。改值同步重算 `KeyHash`（鉴权切到新值）+ 重写 `KeyEnc`（可查看）。运营据此自控这把永久 key（亦是老哈希密钥「改为已知值后即可查看」的恢复路径）。
3. **保留「吊销」+ 强警告**：吊销仍在，但 `DangerConfirm` 文案明确「吊销=使用此密钥的已分发客户端将无法再更新」，避免误吊销已发出整合包所依赖的密钥。

> 改值与轮换的本质差异：轮换是「系统生成一把全新随机 key」（运营不可控其值、易当作无害刷新而误触），改值是「管理员显式设定一个已知值」（运营有意为之、可在改值同时把新值下发玩家）。对一把半公开、永久使用的凭据，后者才是正确的运营心智模型。

## 理由

- **安全级不降，与真实信任级一致**：拉取密钥半公开（随整包分发必泄露），防投毒**全靠 manifest 的 Ed25519 签名**（ADR-022 决策②，**本 ADR 完全不动**）。可查看不触碰签名信任根——私钥仍服务端持有、env 注入、不随分发；攻击者即便拿到明文拉取密钥也只能**拉取**（本就半公开），无法伪造合法签名 manifest、无法投毒。
- **净风险低、可审计**：可查看意味着「DB **且** env 加密密钥同时泄露」才暴露明文 key；而 key 本就随整合包半公开，这点增量暴露面相对其本就半公开的现实可忽略。每次查看写审计 `client_key.reveal` 以便追溯。
- **解决真实运营灾难**：消除「忘记密钥只能轮换→已分发玩家集体断更」的两难。

## 后果

- `ClientPullKey` 加 `KeyEnc` 列（加性迁移，GORM AutoMigrate 自动加列，默认空，老库零改动可用）。
- 新增 env 约定 `JIANMANAGER_CLIENT_KEY_ENC_SECRET`（32 字节 base64）；新增加密工具（`internal/controlplane/service`，AES-256-GCM，未配优雅降级）。
- 新增端点 `GET /client-channels/:id/keys/:keyId/reveal`（平台管理员 + 审计 `client_key.reveal`）；`ARCHITECTURE.md` 数据库 ER `client_pull_keys` 增 `key_enc`、`API.md` 增该端点。
- 鉴权（`VerifyKey`/`VerifyAnyKey` 用 `KeyHash` 比对）**零改动**；`KeyEnc` 不参与鉴权、不序列化给客户端。
- **加密密钥轮换为后续事项**（风险）：若更换 `JIANMANAGER_CLIENT_KEY_ENC_SECRET`，旧 `KeyEnc` 解不开（解密失败按「不可查看」处理，不崩）；首期固定一把，密钥版本化留作后续。

## 替代方案

- **维持只存哈希（现状）** — 造成「忘记密钥→只能轮换→已分发玩家集体断更」的运营灾难，且「只存哈希」对一把半公开凭据并未换来实质安全，否决。
- **明文落库** — 无加密层，DB 单点泄露即暴露所有 key，且不符 JM「敏感信息不裸存」惯例，否决；可逆加密在「可查看」与「DB 泄露防护」间取平衡。
- **把拉取密钥升级为强机密（不可查 + 强存储）** — 与其「随整包分发必半公开」的本质矛盾，是把强安全错配到半公开凭据上（同 ADR-022 决策①理由），否决。
- **可查看但不加密密钥来源化（硬编码一把固定密钥）** — 违反 `config-files` 规范（敏感信息不硬编码生产值），且一把写死的密钥随源码公开即等于明文，否决；故 env 注入 + dev 回退。

## 修订 ADR-022 决策①

ADR-022 决策①原文：「密钥**落库只存 SHA-256 哈希**，明文仅创建 / 轮换时一次性返回、不可二次读取」。

本 ADR **supersede 该决策的存储部分**，改为：「`KeyHash`（SHA-256）用于**鉴权**（鉴权语义不变）；另存 AES-256-GCM 可逆加密副本 `KeyEnc`（env 注入密钥、未配优雅降级）供平台管理员**查看明文**。可查看与拉取密钥『半公开、非信任根』的真实信任级一致；签名（决策②）信任根、私钥持有、防投毒模型**完全不受影响**。」ADR-022 决策①的**鉴权部分**（用哈希比对）与决策②~⑧、状态 `accepted` 均**不变**，仅存储/可查看性被本 ADR 修订。
