# ADR-028: CP 业务数据与时序监控数据分表分策略

- **日期**: 2026-06-24
- **状态**: accepted（通用 envelope + 经济镜像/审计表已随 FR-122 落地：`business_events` / `economy_balance_mirrors` / `economy_ledger_entries`；背包专表随 FR-126 续）
- **上下文**: JBIS 汇聚回 CP 的业务数据（余额变动、背包操作）与既有时序监控数据（指标快照）性质不同：监控是时序、可降采样、可丢（ADR-013 已有降采样策略）；业务是状态 + 审计，要幂等、要留痕、不能丢。架构不变量：数据库仅 CP 可读写，Worker/探针不碰 DB（业务真源仍在各插件存储，JM 存的是汇聚镜像 + 操作审计）。

## 决策

**业务数据与监控数据分表、分持久化策略；业务数据持久可审计、按 `dedupKey` 去重。**

1. **通用 envelope 表**（插件无关汇聚底座）：`domain` / `action` / `payload-json` / `dedupKey` / `node` / `operator` / `ts`，业务事件按 `dedupKey` 幂等落库（至少一次投递去重）。新增域无需改表。
2. **高价值数据可选结构化专表**（按域加性扩展，不替代 envelope）：如经济镜像/流水审计、背包操作审计——供定制页高效查询与对账。
3. **与监控分治**：业务数据**不**降采样、**不**丢；监控延用 ADR-013 raw/5m/1h 降采样。两者不混表、不混策略。
4. **不复制业务真源**：JM 存的是汇聚镜像 + 操作审计，余额/背包真源仍在各插件库（守数据所有权不变量）。

## 理由
- 通用 envelope 让"接新域"零改表（O(1）汇聚）；结构化专表满足核心域高效查询。
- `dedupKey` 去重应对桥事件至少一次投递与跨节点重试。
- 分治避免把"不能丢的业务审计"误用监控的"可降采样可丢"策略。

## 后果
- CP 新增业务 envelope 表 + 按域结构化专表（FR-116 底座 / FR-122 经济 / FR-126 背包）。
- 备份策略需纳入业务表（与监控派生数据 `var/index`/`cache` 的"可丢"取舍不同）。
- 多区/多节点聚合需带 `node→zone` 维度（经济跨区同名玩家不串味，FR-122）。

## 关系
- **ADR-013（监控时序降采样）**：本 ADR 与之分治，业务不走降采样。
- **架构不变量**：DB 仅 CP 读写；业务真源不进 JM DB（只存镜像+审计）。
- **设计总纲** §11 / 建议 ADR-028。
