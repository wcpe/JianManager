# ADR-023: 客户端分发端点的防护与公网暴露——L7 自研 + L3/L4 靠 CDN/云清洗

- **日期**: 2026-06-23
- **状态**: accepted

## 上下文

客户端 OTA（[ADR-021](021-client-distribution-jvm-updater.md) / [ADR-022](022-client-manifest-trust-and-public-endpoint.md)）让 Control Plane 新增一类**面向玩家公网**的拉取端点（manifest / 制品 / 遥测）：规模大、公网可达、是新的攻击面。需求方明确要"防 CC/DDoS、能拒绝某些 IP、支持白名单、清洗正常下载流量"。

必须先厘清一个边界：**容量型（L3/L4）DDoS——打满带宽 / 连接的洪水——应用层（JM 进程内）物理上扛不住**，那是 CDN / Anycast / 云清洗的职责。JM 能有效做的是**应用层（L7）防护**。把这条说清楚，避免"全在 JM 自研就安全"的错误预期。

另一个前提（来自 FR-092）：**机器码由客户端生成、可伪造、不可信**，不能作为限流 / 封禁的主键。

## 决策

1. **JM 做 L7 应用层防护**：限流（**以 IP 为主维度**，机器码仅辅助）、IP 黑名单（直接拒）、白名单模式（仅白名单可访问）、连接数 / 并发限制、异常流量识别（请求突增 / 异常 UA / 无效 key 高频）、可缓存内容走缓存。黑白名单与限流阈值**运行时可改、入审计**。

2. **L3/L4 容量型攻击不在 JM 内**：明示由 CDN / Anycast / 云清洗承接；JM 不做无效承诺、不在应用层空耗资源对抗带宽洪水。

3. **机器码不可信、限流以 IP 为主**：机器码客户端生成可伪造，仅作统计与**辅助**维度；限流 / 封禁主键为 IP。统计口径须容忍机器码污染（去重 / 异常值过滤）。

4. **缓存即防护**：manifest / 制品是静态可缓存内容，前置 CDN 承接绝大多数请求，源站只处理回源——既降载又是最有效的抗量手段。manifest 缓存需版本化 URL / 短 TTL，保证发布后及时可见（与 FR-087 一致）。

5. **公网入口与运营入口隔离**：面向玩家端点用**拉取密钥**鉴权（非浏览器 JWT），与运营者浏览器入口在鉴权与暴露面上隔离。`.claude/rules/architecture-invariants.md` 与 `docs/ARCHITECTURE.md` 在 FR-087/096 落地时增补「面向玩家公网分发端点」为 CP 的第二类入口（端点未实现前不改不变量正文，避免文档超前于代码）。

## 理由

- **诚实分层**：应用层防 CC、基础设施防容量型，各司其职；给运营正确预期而非"全自研可挡 DDoS"的幻觉。
- **IP 为主**：机器码可伪造，以 IP 为限流 / 封禁主键才有实效；机器码只增不减地用于统计与辅助。
- **缓存抗量**：静态分发内容天然可缓存，CDN 是性价比最高的抗量与降载手段，与 FR-087 的可缓存 manifest / 内容寻址制品天然契合。

## 后果

- FR-096 实现 L7 防护（限流 / 黑白名单 / 连接数 / 异常识别 / 缓存）；**运营方须自行配置 CDN / 云清洗**扛 L3/L4（与"客户端整包 / 分发由运营方自理"的范围边界一致）。
- `architecture-invariants.md` / `ARCHITECTURE.md` 在 FR-087 落地时增补面向玩家公网分发入口一类，明确其不持浏览器会话、只读分发、与运营入口隔离。
- 黑白名单 / 限流配置运行时可改并入审计；防护动作可观测（接入既有监控 / 审计）。
- 关联 FR：FR-096（L7 防护主体）、FR-092（机器码不可信、IP 为主）、FR-093（审计追踪）、FR-087（可缓存端点）。

## 替代方案

- **纯 JM 自研抗 DDoS** — 应用层扛不住容量型洪水（带宽 / 连接耗尽），否决；必须配基础设施。
- **不做 L7、全靠 CDN** — CDN 防容量，但挡不住精细 CC、无效 key 高频刷源站、需精细黑白名单的场景，L7 仍不可少，否决。
- **以机器码为限流 / 封禁主键** — 机器码客户端可伪造，作主键即可被绕过，否决；仅作辅助与统计。
