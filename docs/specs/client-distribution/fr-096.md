# 实施计划 — FR-096 分发端点应用层（L7）防护

> 关联 FR: FR-096 | 优先级: P1 | 状态: 🔨 in-progress | 关联 ADR: ADR-023
> 依赖: FR-087（挂消费端点）、FR-092（IP 为主、机器码不可信仅辅助）

## 设计要点（仅 L7；L3/L4 容量型 DDoS 靠 CDN/云清洗，明确不在 JM 内）

- **守卫中间件** `ClientDistGuard` 挂在面向玩家消费端点（manifest/制品）：① IP 黑白名单 ② **per-IP 令牌桶限流**（复用 `RateLimiter`，IP 为主键）③ 全局并发信号量。命中 → 403（IP 拒）/ 429（限流/繁忙），不进 handler。
- **IP 规则**（运行时可改 + 入审计）：`client_ip_rule`（cidr + mode=deny|allow + note）。`ClientIPGuardService` 内存缓存 + CRUD 时重载；`Allowed(ip)`：有任一 allow 规则 → **白名单模式**（仅匹配 allow 的 IP 放行）；否则匹配 deny → 拒。CIDR/单 IP 解析（单 IP 视作 /32、/128）。
- **异常流量**：请求突增 / 无效 key 高频 → 同一 IP 高频自然被 per-IP 限流拦截（无需单独识别 UA，低价值且易误伤）。
- **缓存即防护**：manifest/制品已是可缓存（FR-087 ETag/304 + 内容寻址强缓存），CDN 前置承接绝大多数请求 → 文档强调，源站只回源。
- **可观测且不放大**：守卫用**内存原子计数器**（deny/rate/concurrency 拦截数），经 `GET /client-dist/protection-stats`（admin）暴露——**绝不按被拦请求写库**（否则攻击下自我放大）。
- **运行时可改、入审计**：IP 规则增删经 admin 端点，写 audit（`client_ip_rule.add`/`.remove`）。

## 验收映射
| 验收 | 实现 |
|---|---|
| 限流以 IP 为主 + 端点级速率/突发 | `ClientDistGuard` per-IP 令牌桶（复用 RateLimiter） |
| IP 黑名单（直接拒）+ 白名单模式；运行时可改、入审计 | `client_ip_rule` + `ClientIPGuardService`（缓存重载）+ admin CRUD + audit |
| 连接数/并发限制；异常流量识别 | 全局并发信号量；高频/无效 key 高频经 per-IP 限流拦截 |
| 可缓存内容走缓存 | FR-087 ETag/304 + 内容寻址强缓存（文档强调 CDN 前置） |
| L3/L4 靠 CDN、JM 只做 L7 | 文档明确边界（ADR-023） |
| 防护动作可观测 | 内存计数器 + `GET /client-dist/protection-stats`（不按请求写库防放大） |

## 不做（范围边界）
- L3/L4 容量型 DDoS（带宽/连接洪水）→ CDN/Anycast/云清洗，运营方配置，不在 JM。
- 地理封禁/ASN 规则、UA 指纹识别 → 暂不做（低价值/易误伤）；IP 规则足够。
