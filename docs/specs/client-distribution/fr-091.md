# 实施计划 — FR-091 updater-core 自更新 + 客户端 N-1 回退

> 关联 FR: FR-091 | 优先级: P1 | 状态: 🔨 in-progress | 关联 ADR: ADR-021（纯 JVM、内存加载便于自更新）、ADR-022（签名信任）
> 依赖: FR-090（reconcile/transport/StateStore/CasCache 就绪）
> 纯客户端线：服务端 `PublishVersion` 已支持声明 `manifest.agent.core`（FR-087），核心 jar 走既有制品端点，**无需服务端改动**。

## 背景

楔子（FR-089）以内存 `URLClassLoader` 加载 `updater-core.jar`（不锁原 jar，正为自更新铺路）。FR-091 让 **core 自身**可经 manifest
的 `agent.core` 段自更新到新版，并提供 **N-1 + boot-success 确认 + 启动失败自动回退**，避免坏 core 把玩家锁死在更新阶段。

manifest `agent.core`（契约 §2）：`{ "version": N, "platforms": { "windows|macos|linux": { "artifact": { "sha256, size, codec } } } }`。
core jar 由运营方经 `POST /files` 上传得 sha256、发布版本时填入 `agent.core`，客户端按本机平台经 `/client-artifacts/{sha256}` 下载。

## 设计要点

### 状态与产物（`<gameDir>/.jm-updater/core/`）
- `state.properties`（**java.util.Properties**，wedge 与 core 均 JDK 原生读写、零 JSON 依赖、临时文件+原子 move）：
  `selectedSha/selectedVersion`（wedge 当前应加载的 core）、`prevSha/prevVersion`（N-1）、`pendingSha/pendingVersion`（已下载待首启确认）、`pendingTried`（已被加载过一次）。
- `<sha>.jar`：内容寻址 core jar，**N-1 自然保留**（selected 与 prev 各引用一份；下载零重复）。
- `pending.confirmed`：**boot-success 标志文件**，由看门狗在游戏存活 T 秒后创建。

### 职责分工（与现有边界一致）
- **core（被楔子加载、有 transport/manifest）**：reconcile 成功后比对 `agent.core.version` 与本机运行 coreVersion，更高且尚未 pending →
  下载新 core jar（`transport.fetchArtifact` + codec 解码）→ **sha256 校验**（manifest 已 Ed25519 验签，声明 sha256 可信）→
  **selftest**（throwaway `URLClassLoader` 载新 jar、校验 `Core.run(Map)` 存在 + 调 `Core.selfTest()` 返回 OK）→ 写 `<sha>.jar` + 置 pending。core **不碰** selected/prev。
- **wedge（稳定件、加载方）**：premain 加载前跑**选择状态机**决定加载哪个 core jar + coreVersion；跑完若本次是 trial → 起 **boot-confirm 看门狗** daemon 线程。

### 选择状态机（wedge premain，加载前）
1. 读 state.properties（缺失=全空 → 加载 bundled `config.coreJar`，coreVersion=0）。
2. `pending` 存在时：
   - `pending.confirmed` 标志在 → **promote**：prev=selected，selected=pending，清 pending/tried/confirmed；store。
   - 否则 `pendingTried` 为真 → **判定上次 trial 启动失败（崩溃/未存活到 T）** → **回退**：清 pending/tried（弃用，jar 留 CAS），selected 不变（=N-1）；store；日志告警。
   - 否则（pending 未 tried）→ **首次 trial**：置 `pendingTried`，store，本次加载 **pending**（而非 selected），标记 thisRunIsTrial。
3. 选定 coreJar：`selected`(或 trial 的 pending) 的 `<sha>.jar` 存在则用之，否则 bundled。coreVersion 随之。
4. 加载并 `Core.run(ctx)`（ctx.coreVersion=选定版本）。
5. 若 thisRunIsTrial 且 run 未致命 → 起看门狗 daemon：sleep `bootConfirmSec`（默认 30s），存活则 `touch pending.confirmed`（仅建文件，无读改写竞态）。游戏崩溃则 daemon 随 JVM 死、标志不建 → 下次回退。

> **早退权衡**：玩家 <T 秒退出 → 未建 confirmed → 下次回退并重 trial（CAS 命中、零重下，自愈；某次会话存活过 T 即 promote）。可接受，文档化。

### 手动/运营回退（客户端执行端）
- 运营整体回滚=服务端以更高版本重发（FR-088）。客户端手动回退：放置哨兵文件 `<state>/core/rollback.flag` → 下次 premain 强制弃 pending+selected 回 prev（无 prev 则回 bundled），用后删除 flag。最小支持，覆盖「本地坏更新想立刻退」。

### 不变量守护
- 楔子全程 **fail-open**：选择状态机/看门狗任何异常都 try/catch 后按「加载 bundled」兜底，绝不挡游戏。
- core 自更新失败（下载/验签/selftest 任一）→ 不置 pending、保持现状、本次照常 reconcile 放行（fail-static 语义不变）。
- 原子：state.properties 临时文件+move；jar 先写 `<sha>.jar.tmp` 再 move；pending 仅在 jar 落地+selftest 通过后置。

- [x] `Core`：新增 `public static int selfTest()`（返回固定 OK 码，供新 jar 被 selftest 加载时自证可用）。

## 验收映射（FR-091）

| 验收标准 | 实现 |
|---|---|
| 自更新：声明版本+hash，不符则下新 jar、验签+selftest 后切换、失败回退旧 jar；内存加载避锁、下次 premain 生效 | SelfUpdater 下载+sha256+selftest 置 pending；wedge 下次 premain promote/load |
| N-1 保留（CAS 零额外全量） | `<sha>.jar` 内容寻址，selected+prev 各引用、下载去重 |
| boot-success 确认（hook 或超时无崩溃）+ 下次未确认→回退 N-1 | 看门狗存活 T 建 `pending.confirmed`；premain 据 confirmed/tried 判 promote/rollback |
| 手动/运营回退（客户端执行端） | `rollback.flag` 哨兵；运营整体回滚见 FR-088 |
| 原子、不损坏、可重入 | properties/jar 临时文件+move；pending 仅 selftest 通过后置；SingleInstanceLock（reconcile 既有） |
| 真机：推新版→下次生效；坏版本→启动失败→下次自动回退 N-1 | 真机脚本：发 agent.core 新版→次启 promote；注入坏 core→trial 崩→次启 rollback |

## 不做（范围边界）
- 游戏内 hook 式 boot-confirm（需 mod）→ 采用「超时无崩溃」看门狗即可满足验收。
- 楔子（wedge）自更新 → 楔子随基础包低频分发、设计为稳定件（contract §2 agent.wedge 仅信息性），不自更新。
- core 多级历史（N-2…）→ 仅保 N-1。
